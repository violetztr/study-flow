# 面试展示讲法

这份文档用于面试或项目复盘。目标不是背稿，而是帮你把 Ruru 社区讲成一个有技术含量、有产品思考、能继续扩展的全栈项目。

## 30 秒介绍

Ruru 社区是一个基于 Spring Boot、React、MySQL、Redis、Docker 和 Cloudflare R2 的 B 站式小圈子社区平台。

它支持图文和视频内容、视频封面、投稿审核、评论、弹幕、点赞、投币、收藏、关注、用户主页、播放统计、观看历史和管理员内容治理。媒体文件不走业务服务器中转，而是通过后端签发预签名 URL，由浏览器直传到对象存储。当前是单体架构，但模块边界已经按用户、内容、媒体、互动、审核来组织，后续可以继续演进到微服务。

## 项目亮点

- 不是普通 CRUD：包含视频投稿、审核、弹幕、播放统计、观看历史和对象存储。
- 有真实部署：Docker Compose 部署在云服务器，Nginx 反向代理到线上域名。
- 有媒体架构：图片和视频上传到 Cloudflare R2，服务器只保存元数据，避免占用服务器磁盘和带宽。
- 有内容治理：普通用户投稿，管理员审核通过后公开视频；管理员也能删除帖子、评论、弹幕。
- 有数据建模：MySQL 用 Flyway 管理迁移，拆分用户、帖子、媒体、弹幕、关注、收藏、观看历史等表。
- 有安全意识：JWT 鉴权、R2 密钥只在后端、Bucket 私有、前端只拿预签名 URL。
- 有测试保障：后端接口和迁移测试覆盖核心行为，前端使用 TypeScript 构建和 lint 检查。

## 架构怎么讲

```text
React 前端
  -> /api
  -> Spring Boot 后端
  -> MySQL: 用户、帖子、评论、弹幕、观看历史
  -> Redis: 当前用于基础设施，后续可做限流、缓存、计数
  -> Cloudflare R2: 图片、视频、封面对象存储
```

前端负责页面体验和直传 R2；后端负责权限、业务规则、数据库事务和预签名 URL；MySQL 保存结构化业务数据；R2 保存大文件；Docker Compose 把前端、后端、MySQL、Redis 组合部署。

## 关键业务流程

### 视频投稿流程

1. 用户选择视频和封面。
2. 前端请求后端获取 R2 预签名上传 URL。
3. 前端直接 PUT 到 R2。
4. 上传完成后通知后端，后端记录媒体状态。
5. 用户提交稿件，视频帖子进入 `PENDING_REVIEW`。
6. 管理员审核通过后，帖子变为 `PUBLISHED`，视频进入公开列表。

### 播放量流程

1. 打开详情页不直接增加播放量。
2. 前端监听播放器进度。
3. 播放超过 10 秒或超过 20% 后，向后端上报。
4. 后端用用户 ID 或游客标识做去重。
5. 同一个用户重复播放只更新观看历史，不重复加播放量。

### 弹幕流程

1. 弹幕绑定视频时间点 `timeSeconds`。
2. 后端按时间顺序返回弹幕。
3. 前端播放到对应秒数后让弹幕飘过。
4. 用户可以关闭弹幕和选择颜色。
5. 管理员可以删除不合适的弹幕。

## 为什么不用服务器存视频

视频和图片属于大文件，如果直接存在服务器，会占用磁盘、消耗带宽，也不方便以后接 CDN。对象存储更适合这类文件。

当前设计是：

- 业务服务器只保存 `object_key`、文件类型、大小、上传人和状态。
- 文件本体放 Cloudflare R2。
- Bucket 保持私有，读写都通过后端生成有时效的签名 URL。

这比把文件放到 Spring Boot 服务或服务器硬盘里更接近真实生产方案。

## 为什么现在还没上微服务

当前阶段先做单体，因为项目还在验证核心业务闭环。单体更容易开发、部署、调试和测试。

但是代码和表已经按模块划分：

- 用户模块：账号、资料、关注、钱包。
- 内容模块：帖子、图文、视频、投稿状态。
- 媒体模块：R2 上传、媒体元数据。
- 互动模块：评论、点赞、投币、收藏、弹幕。
- 审核模块：投稿审核和管理操作记录。

以后如果流量和复杂度上来，最适合先拆的是媒体服务、审核服务、互动计数服务和搜索推荐服务。

## 面试官可能会问

### Q：这个项目和普通博客/论坛有什么区别？

普通论坛重点是帖子和评论。Ruru 社区增加了视频投稿审核、对象存储上传、弹幕、播放统计、观看历史、投币和内容治理，更接近内容平台。

### Q：播放量为什么不等于详情页访问量？

因为打开页面可能只是误点或刷新，不代表真实观看。现在只有播放超过 10 秒或超过 20% 才计入一次播放，并且同一个用户/游客会去重。

### Q：弹幕为什么放 MySQL，不放 MongoDB？

当前数据量小，MySQL 足够，而且便于事务和项目部署。后期如果弹幕量变大，可以迁到 MongoDB 或按视频 ID 分表，也可以用 WebSocket 做实时弹幕。

### Q：Redis 现在用得多吗？

当前 Redis 已经在 Docker 架构里准备好，后续会用于登录会话扩展、限流、防刷、热点计数、排行榜和缓存。当前阶段先保证业务闭环和可部署。

### Q：如何防止恶意上传大视频？

当前已经做了基础限制：后端签发上传 URL 前检查文件类型和大小，R2 密钥不暴露给浏览器，视频投稿需要管理员审核后才公开。后续还会加用户配额、上传频率限制、内容安全审核和转码队列。

## 下一阶段可以怎么做

- FFmpeg 异步转码：原视频上传后生成 720P、1080P 和 HLS 切片。
- Redis 限流：防刷播放、点赞、投币和上传接口。
- 消息队列：视频上传完成后异步转码、异步审核、异步通知。
- 搜索系统：接 Elasticsearch 做标题、简介、话题和作者搜索。
- 推荐系统：先做热门、最新、关注、标签匹配，再做行为推荐。
- 直播模块：SRS 或 Nginx-RTMP，支持直播间、聊天和实时弹幕。
- 监控告警：Prometheus + Grafana 观察接口、容器和数据库状态。
