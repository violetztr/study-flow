# Ruru 生产级路线图

这份文档是 Ruru 社区从“能跑起来的小项目”升级到“可以真实上线、能展示技术深度、后续可拆微服务的大项目”的总路线。

它不是一次性全做完的幻想清单，而是后续开发的主线：每次继续推进时，都从这里选一个阶段，拆成详细任务，写测试，开发，实现，部署，再进入下一阶段。

## 1. 项目最终定位

Ruru 要做成一个小圈子内容社区，产品形态优先参考 B 站，而不是抖音。

核心方向：

- 图文内容：适合朋友动态、帖子、图片分享、长文说明。
- 视频内容：适合投稿、封面、审核、播放页、弹幕、评论、点赞、投币、收藏。
- 直播内容：后期接入直播间、实时聊天、直播弹幕和在线人数。
- 社交关系：关注、粉丝、用户主页、个人资料、创作者内容列表。
- 内容治理：投稿审核、管理员删除违规内容、评论和弹幕管理。
- 平台能力：缓存、限流、搜索、推荐、消息队列、转码、监控、备份、CI/CD。

一句话目标：

> Ruru 不是普通 CRUD 社区，而是一个从内容发布、媒体上传、审核、播放、互动、治理到部署运维都有完整链路的内容平台。

## 2. 当前项目状态

当前 Ruru 已经具备一个“内容平台雏形”：

- 前端：React、TypeScript、Vite、Ant Design、React Router、TanStack Query。
- 后端：Java 17、Spring Boot 3、Spring Security、JWT、MyBatis-Plus、Flyway。
- 数据库：MySQL 8 保存用户、帖子、评论、弹幕、关注、收藏、播放历史等业务数据。
- 缓存基础：Redis 已经在 Docker Compose 中准备好，后续用于限流、缓存、计数、排行榜。
- 媒体存储：Cloudflare R2 保存图片、视频和封面，业务服务器只保存元数据。
- 部署：Docker Compose 部署在云服务器，Nginx 反向代理到 `https://www.violet-surf.com`。
- 已完成核心功能：注册登录、图文、视频投稿、封面、审核、播放详情、弹幕、评论、点赞、投币、收藏、关注、播放历史、管理员治理。

当前最大短板：

- 还没有 Redis 限流和热点缓存。
- 视频还没有 FFmpeg 转码和 HLS 分片。
- 互动计数还主要依赖数据库，抗高并发能力有限。
- 还没有消息队列做异步任务。
- 还没有搜索、推荐、榜单。
- 还没有系统监控、日志聚合、备份和自动部署流水线。
- 目前还是单体应用，尚未真正拆成微服务。

## 3. 总体架构演进原则

### 3.1 先单体，后微服务

现在不要急着拆微服务。原因很现实：

- 当前用户量小，单体更容易开发、测试、部署和排查问题。
- 业务边界还在变化，太早拆服务会让你把精力浪费在服务通信、配置、注册中心和部署复杂度上。
- 微服务不是高级的代名词，能把单体写清楚、模块边界设计好，才是以后拆服务的基础。

正确路线：

1. 先把单体做成清晰模块化单体。
2. 把高并发、缓存、异步、转码、搜索、监控这些生产能力补齐。
3. 当某个模块真的有独立扩展需求时，再拆微服务。

### 3.2 数据库先稳定，再做冷热分离

MySQL 继续作为主库，保存强业务关系数据：

- 用户
- 内容
- 评论
- 弹幕元数据
- 审核状态
- 互动关系
- 播放历史
- 钱包和每日奖励

Redis 用来解决速度和并发问题：

- 限流
- 热点内容缓存
- 播放防刷
- 点赞、投币、收藏计数
- 在线人数
- 排行榜
- 临时上传凭证

MongoDB、Elasticsearch、Kafka 这类组件后面再接，不要为了“看起来高级”过早引入。

### 3.3 媒体永远不放业务服务器

图片、视频、封面继续放对象存储。业务服务器只保存：

- 文件 key
- 文件类型
- 文件大小
- 上传人
- 状态
- 审核结果
- 转码结果

后续视频播放会演进成：

```text
用户上传原视频
  -> R2 pending 区
  -> 后端记录稿件
  -> 审核通过
  -> FFmpeg 转码
  -> 生成 HLS 分片和不同清晰度
  -> R2 published 区
  -> 前端播放器播放 HLS
```

## 4. 阶段路线总览

推荐按下面顺序做：

```text
阶段 1：当前版本稳定化
阶段 2：安全和权限强化
阶段 3：Redis 高并发基础
阶段 4：视频转码和 HLS 播放
阶段 5：消息队列和异步任务
阶段 6：搜索、推荐和榜单
阶段 7：直播基础
阶段 8：监控、日志、备份和 CI/CD
阶段 9：模块化单体升级
阶段 10：微服务拆分
阶段 11：生产级高可用
```

后续每个阶段都要满足一个原则：

> 每个阶段都必须能上线、能测试、能解释、能写进简历。

## 5. 阶段 1：当前版本稳定化

目标：

把已有功能从“能用”变成“稳定、好看、好维护”。

要做的事：

- 清理前端冗余 UI 和无用文案，保持简洁内容平台风格。
- 统一图文、视频、直播频道的数据结构和页面交互。
- 优化视频卡片、详情页、评论区、弹幕区、管理员后台体验。
- 修复刷新、切换频道、图片加载、视频封面加载的体验问题。
- 明确所有接口的错误提示，让用户知道是没登录、没权限、审核中还是系统错误。
- 补齐关键 README、接口文档、数据库文档、部署文档。

验收标准：

- 线上首页、图文、视频详情、评论、弹幕、投稿、审核、登录注册都稳定可用。
- 页面没有明显多余文案和土味布局。
- 新用户打开网站能知道这是一个内容社区。
- 管理员能处理投稿、评论、弹幕和违规内容。
- 后端测试、前端构建、lint 都通过。

你要学会：

- React 页面拆分。
- 前端状态管理和接口请求。
- Spring Boot Controller、Service、Mapper 分层。
- MySQL 表结构如何支撑业务。
- Docker Compose 如何把服务跑起来。

## 6. 阶段 2：安全和权限强化

目标：

让网站不会被随便刷接口、刷上传、乱删数据。

要做的事：

- 明确权限模型：游客、普通用户、管理员 ruru。
- 所有写操作必须登录：发布、评论、点赞、投币、收藏、发弹幕、关注。
- 管理操作必须管理员权限：审核、删除帖子、删除评论、删除弹幕、封禁用户。
- 登录注册接口增加基础频率限制。
- 上传接口增加大小、类型、次数限制。
- R2 上传 URL 保持短有效期。
- 管理员操作全部记录日志。
- 前端根据权限隐藏不能操作的按钮，但后端必须再次校验权限。

验收标准：

- 游客可以浏览公开内容，但不能互动。
- 普通用户不能访问管理员接口。
- 管理员可以治理全站内容。
- 恶意重复请求不会轻易打爆登录、上传、评论接口。
- R2 密钥不会出现在前端代码和 GitHub。

你要学会：

- Spring Security 权限控制。
- JWT 鉴权流程。
- 后端权限不能只靠前端隐藏按钮。
- 常见接口攻击方式。

## 7. 阶段 3：Redis 高并发基础

目标：

让项目开始具备真实平台的抗并发能力。

优先做推荐方案：

- Redis 限流。
- 播放量防刷缓存。
- 热点 feed 缓存。
- 视频详情缓存。
- 互动计数缓存雏形。

要做的事：

- 建立 Redis key 命名规范。
- 给登录、注册、上传、评论、弹幕、点赞、投币、播放上报加限流。
- 播放上报先查 Redis，避免频繁写数据库。
- 首页 feed 和视频详情做短 TTL 缓存。
- 发布、审核、删除、评论、点赞后清理对应缓存。
- Redis 异常时降级到 MySQL，不让网站直接挂掉。

推荐 key 示例：

```text
rate:login:{ip}
rate:upload:{userId}
rate:comment:{userId}
view:dedupe:{postId}:{viewerKey}
feed:{contentType}:{page}
post:detail:{postId}
counter:post:{postId}
```

验收标准：

- 同一 IP 短时间疯狂登录会被限制。
- 同一用户短时间疯狂评论、发弹幕会被限制。
- 同一视频重复刷新不会重复增加播放量。
- 首页和视频详情重复访问时数据库压力下降。
- Redis 挂掉时核心页面仍能尽量访问，只是性能下降。

你要学会：

- Redis String、Hash、Set、ZSet 的使用场景。
- TTL 过期时间。
- 缓存穿透、缓存击穿、缓存雪崩。
- 限流算法的基本思路。

## 8. 阶段 4：视频转码和 HLS 播放

目标：

从“直接播放用户上传的 MP4”升级到“真正视频平台播放链路”。

要做的事：

- 后端记录视频原文件、转码状态、转码结果。
- 引入 FFmpeg。
- 审核通过后触发转码任务。
- 生成 720P、1080P 等清晰度。
- 生成 HLS 切片文件。
- 前端播放器支持 HLS 播放。
- 播放页显示清晰度选择。
- 转码失败要能记录原因，并允许管理员重新转码。

推荐流程：

```text
视频上传完成
  -> post 状态 PENDING_REVIEW
  -> 管理员审核通过
  -> media 状态 TRANSCODING
  -> FFmpeg 生成 HLS
  -> 上传 HLS 文件到 R2
  -> media 状态 READY
  -> 前端播放 HLS master.m3u8
```

验收标准：

- 用户上传原视频后，公开播放的是转码后的 HLS。
- 播放器可以播放分片视频。
- 视频可以按清晰度播放。
- 转码失败不会影响整个系统。
- 原视频和转码后文件路径清晰区分。

你要学会：

- FFmpeg 基础命令。
- MP4 和 HLS 的区别。
- 视频分辨率、码率、帧率。
- 为什么大平台不直接播放原视频。

## 9. 阶段 5：消息队列和异步任务

目标：

把耗时任务从用户请求里拆出来，避免接口卡住。

优先选择：

- RabbitMQ：更容易上手，适合当前阶段。
- Kafka：更适合海量日志和推荐行为流，后期再考虑。

要做的事：

- 引入 RabbitMQ。
- 上传完成后发送转码任务消息。
- 审核通过后发送转码消息。
- 评论、点赞、投币可以异步写通知。
- 失败任务进入重试机制。
- 管理后台能看到任务状态。

验收标准：

- 用户提交视频后接口很快返回，不等待转码完成。
- 转码任务由后台消费者处理。
- 转码失败可重试。
- 消息队列挂掉时，系统能提示任务暂时不可用，而不是静默失败。

你要学会：

- 为什么要异步。
- 生产者、消费者、队列、交换机。
- 消息重试和死信队列。
- 幂等性：同一条消息重复消费不能造成脏数据。

## 10. 阶段 6：搜索、推荐和榜单

目标：

让内容可以被找到、被推荐、被排序。

要做的事：

- 先做 MySQL 简单搜索：标题、简介、话题、作者。
- 再接 Elasticsearch 做全文搜索。
- Redis ZSet 做热门榜单。
- 推荐先做规则，不做复杂 AI：
  - 最新
  - 热门
  - 关注的人
  - 同话题
  - 点赞/投币/收藏权重
  - 最近播放完成度
- 记录用户行为：曝光、点击、播放、点赞、收藏、关注。

验收标准：

- 可以搜索视频和图文。
- 首页有热门榜单。
- 用户主页、话题页、搜索页体验完整。
- 推荐逻辑能解释清楚，不是随机展示。

你要学会：

- Elasticsearch 倒排索引。
- Redis ZSet 排行榜。
- 推荐系统从规则开始，而不是一上来深度学习。
- 用户行为数据如何反哺推荐。

## 11. 阶段 7：直播基础

目标：

把直播频道从占位升级成可演示的直播模块。

要做的事：

- 研究 SRS 或 Nginx-RTMP。
- 支持 OBS 推流。
- 支持 HTTP-FLV 或 HLS 拉流。
- 新增直播间表。
- 新增直播状态：未开播、直播中、已结束。
- 直播间支持聊天和弹幕。
- Redis 统计在线人数。

验收标准：

- ruru 可以开一个测试直播间。
- 前端可以观看直播。
- 直播间能发聊天或弹幕。
- 页面显示在线人数。

你要学会：

- RTMP 推流。
- FLV / HLS 拉流。
- WebSocket 实时消息。
- 直播和点播的区别。

## 12. 阶段 8：监控、日志、备份和 CI/CD

目标：

从“能部署”升级到“能稳定运营”。

要做的事：

- 增加健康检查接口。
- Prometheus 采集后端、MySQL、Redis、容器指标。
- Grafana 做监控面板。
- 日志按请求链路输出 traceId。
- 每天自动备份 MySQL。
- R2 文件和数据库记录做一致性检查。
- GitHub Actions 自动运行测试和构建。
- main 分支通过后再部署服务器。

验收标准：

- 服务挂了能看出来。
- 接口慢了能看出来。
- 数据库有备份。
- 每次提交 GitHub 都自动跑测试。
- 部署步骤可重复，不靠手感。

你要学会：

- 什么是可观测性。
- 日志、指标、追踪的区别。
- 为什么线上必须有备份。
- CI/CD 如何减少人为失误。

## 13. 阶段 9：模块化单体升级

目标：

为以后拆微服务做准备，但仍保持单体部署。

要做的事：

- 按业务边界整理包结构：
  - user
  - community
  - media
  - interaction
  - moderation
  - notification
  - search
  - live
- 每个模块只暴露必要接口。
- 模块之间不要乱查对方表，尽量通过 Service 调用。
- 把公共异常、响应、权限、审计、限流抽成 common/infrastructure。
- 为每个模块补独立测试。

验收标准：

- 新人看包结构能知道每个模块负责什么。
- 某个模块可以单独读懂。
- 后续拆服务时，知道先拆哪个模块。

你要学会：

- 模块化单体。
- 领域边界。
- 为什么代码结构比技术名词更重要。

## 14. 阶段 10：微服务拆分

目标：

当业务复杂度真的上来后，把单体拆成多个服务。

不要一开始就全拆。推荐顺序：

1. media-service：媒体上传、转码、R2、HLS。
2. interaction-service：点赞、投币、收藏、评论计数。
3. search-service：搜索索引和搜索接口。
4. notification-service：消息通知。
5. live-service：直播间、聊天、在线人数。
6. user-service：用户、资料、关注关系。

可能技术：

- Spring Cloud Gateway 或 Go 网关。
- Nacos / Consul 做服务注册发现。
- Nacos / Apollo 做配置中心。
- OpenFeign / HTTP / gRPC 做服务通信。
- SkyWalking / Jaeger 做链路追踪。

验收标准：

- 至少拆出一个 media-service，并能独立部署。
- 主服务通过接口调用 media-service。
- 服务挂掉时有降级处理。
- 日志能追踪一次请求经过哪些服务。

你要学会：

- 微服务解决什么问题。
- 微服务带来哪些复杂度。
- 服务注册、配置中心、网关、链路追踪。
- 为什么拆服务前必须先有清晰模块边界。

## 15. 阶段 11：生产级高可用

目标：

让项目具备更接近真实商业系统的稳定性。

要做的事：

- MySQL 定期备份和恢复演练。
- Redis 持久化和内存淘汰策略。
- Nginx 配置 HTTPS、安全响应头、限流。
- CDN 接入静态资源和媒体分发。
- R2 防盗链和签名 URL。
- 上传额度和用户配额。
- 内容审核接入第三方接口。
- 敏感词过滤。
- 管理后台增加操作审计。
- 灰度发布和回滚策略。

验收标准：

- 有备份，有恢复方案。
- 有监控，有告警。
- 有限流，有审核，有权限。
- 有部署文档，有回滚方案。
- 出问题时能定位，而不是靠猜。

你要学会：

- 线上系统不是只写代码。
- 安全、稳定、成本、可维护性都属于工程能力。
- 高级工程师要能对系统负责。

## 16. 现实技术优先级

下面是推荐学习顺序，不要同时学一堆：

### 第一优先级：马上能用于项目

- Spring Boot 深入
- MySQL 表设计和索引
- Redis 限流、缓存、计数
- Docker Compose
- Nginx 反向代理
- Cloudflare R2 / 对象存储
- FFmpeg 基础
- React + TypeScript

### 第二优先级：项目变大后接入

- RabbitMQ
- Elasticsearch
- WebSocket
- Prometheus + Grafana
- GitHub Actions
- HLS 播放器

### 第三优先级：真正需要微服务时再学

- Spring Cloud Gateway
- Nacos / Consul
- Apollo / Nacos Config
- SkyWalking / Jaeger
- Kubernetes
- Kafka
- MongoDB
- 分库分表

### 暂时不要深钻

- 深度学习推荐算法
- 大规模分布式存储自研
- 自研播放器内核
- Kubernetes 复杂集群
- 多活架构

这些不是不重要，而是当前阶段投入产出比不高。

## 17. 后续每次开发的固定流程

以后每做一个阶段，都按这个节奏：

1. 先写阶段设计文档。
2. 再写详细开发任务清单。
3. 每个任务先写测试。
4. 写最小实现。
5. 本地验证。
6. 提交 GitHub。
7. 服务器 `git pull`。
8. `sudo docker compose up -d --build` 部署。
9. 线上验证。
10. 更新文档。

这就是从小白项目变成工程项目的关键：不是一次写很多代码，而是每一步都可验证、可回滚、可解释。

## 18. 下一步建议

当前最适合做的是阶段 3：Redis 高并发基础。

原因：

- Redis 已经在 Docker Compose 里存在，不需要额外买服务。
- 能明显提升项目技术含金量。
- 面试很好讲：限流、防刷、缓存、计数、降级。
- 不会像 FFmpeg、消息队列、微服务那样一下子把复杂度拉爆。

阶段 3 推荐拆成 5 个小任务：

1. Redis key 规范和基础封装。
2. 登录、注册、上传、评论、弹幕、点赞、投币接口限流。
3. 播放量防刷从数据库去重升级为 Redis TTL 去重。
4. 首页 feed 和视频详情短缓存。
5. 点赞、收藏、投币计数缓存雏形。

完成阶段 3 后，项目会从“功能型社区”升级成“开始考虑高并发和生产稳定性的社区”。

## 19. 简历和面试表达

这个项目后续可以这样讲：

> 我做了一个 B 站风格的小圈子内容社区，支持图文、视频投稿、封面、审核、弹幕、评论、点赞、投币、收藏、关注、播放历史和管理员治理。媒体文件使用 Cloudflare R2 对象存储，后端只负责签发预签名 URL 和保存元数据。项目通过 Docker Compose 部署在云服务器，Nginx 反向代理到线上域名。后续我按生产级路线继续接入 Redis 限流和缓存、FFmpeg HLS 转码、消息队列异步任务、搜索推荐、监控告警和微服务拆分。

如果面试官问“你这个项目和普通博客有什么区别”，重点回答：

- 有视频投稿和审核链路。
- 有对象存储，不把大文件压在业务服务器。
- 有播放行为统计，不是简单浏览量。
- 有弹幕、投币、收藏、关注等内容平台互动。
- 有管理员治理和审核。
- 有后续清晰的生产级演进路线。

## 20. 总结

Ruru 的正确成长路线不是一口气堆满所有技术，而是：

```text
先把业务闭环做好
  -> 再把安全和权限做好
  -> 再加 Redis 扛并发
  -> 再加 FFmpeg 做专业视频链路
  -> 再加 MQ 做异步
  -> 再加搜索、推荐、榜单
  -> 再加监控、备份、自动部署
  -> 最后按模块边界拆微服务
```

这条路现实、可学、可上线，也能体现高级全栈工程师的能力。
