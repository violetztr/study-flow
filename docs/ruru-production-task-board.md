# Ruru 生产级任务表

这份任务表是 Ruru 后续开发的执行清单。路线图负责说明“为什么这么做”，任务表负责说明“接下来具体做什么”。

使用方式：

- 状态为 `DONE`：已经完成。
- 状态为 `NEXT`：下一步优先做。
- 状态为 `TODO`：后续阶段做。
- 状态为 `LATER`：等项目规模上来再做。

每次开发只拿一个阶段或一个小批次，不要同时开太多坑。每个阶段完成后都要更新文档、跑测试、提交 GitHub、部署服务器。

## 阶段 3：Redis 高并发基础

状态：`NEXT`

目标：

把项目从“功能能跑”升级成“开始具备抗刷、缓存和热点数据处理能力”。

### 3.1 Redis key 规范和基础封装

状态：`DONE`

任务：

- [x] 设计 Redis key 命名规范。
- [x] 新增 Redis key 常量类，避免代码里到处手写字符串。
- [x] 新增 Redis 工具服务，封装 get、set、delete、increment、expire。
- [x] Redis 出错时记录日志，不能直接让核心业务挂掉。
- [x] 更新文档说明 Redis key 设计。

验收标准：

- Redis key 命名统一。
- 后续功能不直接散落写 Redis 代码。
- Redis 临时异常不会导致首页直接 500。

学习点：

- Redis String、Hash、Set、ZSet 的基本用途。
- TTL 是什么。
- 为什么 key 需要规范。

### 3.2 接口限流

状态：`DONE`

任务：

- [x] 登录接口限流，防止暴力破解。
- [x] 注册接口限流，防止刷账号。
- [x] 上传预签名接口限流，防止恶意刷 R2 上传 URL。
- [x] 评论接口限流，防止刷屏。
- [x] 弹幕接口限流，防止刷弹幕。
- [x] 点赞、投币、收藏接口限流，防止脚本刷互动。
- [x] 播放上报接口限流，防止刷播放量。
- [x] 返回清晰错误提示：请求太频繁，请稍后再试。

验收标准：

- 同一 IP 或同一用户短时间重复请求会被限制。
- 正常使用不受明显影响。
- 被限流时返回 HTTP 429。

学习点：

- 固定窗口限流。
- 滑动窗口限流。
- IP 限流和用户限流的区别。
- 为什么登录接口必须限流。

### 3.3 播放量防刷升级

状态：`DONE`

任务：

- [x] 播放上报达到有效播放条件后，先写 Redis 去重 key。
- [x] 同一用户或同一游客短时间重复播放不重复加播放量。
- [x] 达到有效播放条件后再更新 MySQL 计数状态；普通进度记录仍用于观看历史。
- [x] 已计数的播放在 Redis 中保留 6 小时，减少重复计数压力。
- [x] 保留 MySQL 作为最终播放历史来源。

验收标准：

- 刷新视频详情不会增加播放量。
- 反复拖动或重复上报不会重复计数。
- 播放历史仍然能正常显示。

学习点：

- 为什么播放量不能等于页面访问量。
- Redis 防刷和 MySQL 记录的分工。
- 数据最终一致性。

### 3.4 首页 feed 和视频详情短缓存

状态：`NEXT`

任务：

- [ ] 首页 feed 增加短 TTL 缓存。
- [ ] 视频详情增加短 TTL 缓存。
- [ ] 区分游客缓存和登录用户个性化字段。
- [ ] 不长期缓存 R2 签名 URL，避免 URL 过期后前端拿到坏链接。
- [ ] 发布、审核、删除、评论、点赞、投币、收藏后清理相关缓存。

验收标准：

- 重复刷新首页时数据库查询减少。
- 重复打开视频详情时数据库查询减少。
- 用户点赞、评论后页面数据不会长时间错误。
- R2 签名 URL 不因为缓存导致过期不可用。

学习点：

- 缓存穿透。
- 缓存击穿。
- 缓存雪崩。
- 缓存失效为什么比缓存本身更难。

### 3.5 互动计数缓存雏形

状态：`NEXT`

任务：

- [ ] 点赞数、收藏数、投币数、播放数建立 Redis 计数缓存。
- [ ] MySQL 仍然作为最终可信数据。
- [ ] 写操作后同步更新 Redis 计数。
- [ ] 缓存丢失时从 MySQL 回源。
- [ ] 暂时不做异步批量落库，留到消息队列阶段。

验收标准：

- 首页和详情页读取计数更快。
- Redis 丢失数据可以从 MySQL 恢复。
- 不因为缓存导致点赞数、收藏数明显错乱。

学习点：

- 热点计数。
- 读多写少和写多读多的区别。
- 为什么计数不能一开始就完全依赖 Redis。

## 开发纪律

每做一批任务都要遵守：

- 先理解，再设计。
- 先写测试，再写实现。
- 小步提交，不混着改。
- 任何上线行为都要能回滚。
- 不把真实密钥提交到 GitHub。
- 不为了显得高级而提前堆技术。
- 当前阶段用不到的复杂技术先不接。

如果不知道下一步做什么，就看本文件的“当前优先级”。
